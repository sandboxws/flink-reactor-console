# dashboard/src/routes/hub — Hub Route Tree

Parallel route tree for the **FlinkReactor Hub** redesign, mounted alongside legacy routes. This tree owns its own layout (`<HubShell>`), shares the existing data layer (Zustand stores, urql GraphQL, types), and renders the new visual language from `console-v2/`.

**The legacy tree is unchanged.** Both trees ship side-by-side until cutover (`fr-console-hub-cutover`), at which point `__root.tsx` redirects `/` → `/hub/*` and the legacy routes are deleted.

## How requests flow into this tree

```
/hub/anything
  → routeTree.gen.ts (auto-generated by TanStack Router plugin)
  → __root.tsx detects pathname.startsWith("/hub/") and skips legacy <Shell>
  → /hub/* route component renders <Outlet /> wrapped in <HubShell>
```

The legacy `<Shell>` is **never** in the rendered tree for Hub paths. This is enforced at `dashboard/src/routes/__root.tsx`.

## Hard rules for Hub routes

### 1. Every Hub route MUST wrap its content in `<HubShell>`

There is no nested layout route in this directory yet (a `route.tsx` layout file would be a future refactor). Each route file is responsible for rendering `<HubShell>` itself.

```tsx
import { HubShell } from "@flink-reactor/ui"
import { HUB_SIDEBAR_SECTIONS, HubLink } from "../path/to/shared-helpers"

function MyHubPage() {
  return (
    <HubShell
      sidebarSections={HUB_SIDEBAR_SECTIONS}
      activePath="/hub/my-page"
      LinkComponent={HubLink}
      topBar={{ cluster: { name: "...", env: "prod" } }}
    >
      {/* page content */}
    </HubShell>
  )
}
```

Forgetting this means the page renders bare on `<Outlet />` with no chrome.

### 2. The sidebar sections MUST come from a single canonical source

Today the canonical sidebar lives inline in `sandbox.tsx` as `HUB_SIDEBAR_SECTIONS`. When the second Hub route lands, **extract this constant** to `dashboard/src/lib/hub-sidebar-sections.ts` (or similar). Do not let multiple copies drift.

The sidebar matches `console-v2/overview.html` exactly: 8 sections (Overview / Jobs / Cluster / Observe / Logs / Data / Tools / Instruments) with specific icons, counts, and instrument health glyphs (✓/⚠).

### 3. Use the TanStack Router `<Link>` adapter, not raw `<a>`

`<HubShell>` accepts a `LinkComponent` prop that all sidebar items and the brand link route through. Pass an adapter that wraps TanStack Router's `Link`:

```tsx
import { Link } from "@tanstack/react-router"

function HubLink({
  to,
  className,
  children,
}: {
  to: string
  className?: string
  children: React.ReactNode
}) {
  return (
    <Link to={to} className={className}>
      {children}
    </Link>
  )
}
```

Without this, sidebar clicks trigger a full page reload, losing store state.

### 4. Hub is dark-only in v1

Force dark mode on mount; do not respect the user's `data-palette` preference until the follow-up `fr-console-hub-light-theme` and `fr-console-hub-tokyo-night` changes ship.

```tsx
useEffect(() => {
  document.documentElement.classList.add("dark")
  document.documentElement.removeAttribute("data-palette")
}, [])
```

### 5. `<PageWithRail>` must never have an empty rail

When using a layout with the right rail, the rail content must be useful even before any selection happens. The default rail content is a "summary" view (capacity meters, last deployment, recent checkpoints). When a user selects something (e.g., an operator in the DAG), the rail switches to the rich detail. **Never render an empty rail** — that's the whitespace-gap bug from the original screenshot the user flagged.

If a route doesn't have meaningful rail content, use `HubShell` *without* the `rail` prop (the layout switches to `.page-grid` without `with-rail`).

## Patterns

### Adding a new Hub page

1. **Decide layout**: standard (no rail), with-rail, or full-bleed (sandbox/editor, sql-explorer, kanban).
2. **Match the mockup**: open `console-v2/<page>.html` and read it end-to-end before writing TSX.
3. **Reuse atomic primitives**: `KpiCard`, `LiveDot`, `StatusIcon`, `PriorityBars`, `SevBadge`, `StatePill`, `PropChip`, `HeatmapCell`, `HeatmapCalendar`, `DiffLine`, `DiffViewer`, `ClusterSelector`. All exported from `@flink-reactor/ui`.
4. **For CSS-only primitives**, use the class names directly: `.glass-card-static`, `.activity-entry`, `.kanban-col`, `.log-viewer`, `.code-viewer`, `.tk-*` (syntax tinting), `.resource-bar`, `.sparkbar`, `.health-ring`, `.add-card`.
5. **Wire data**: read from existing Zustand stores (`useClusterStore`, `useAlertsStore`, etc.) — same APIs as legacy routes.

### Hub-only icon sources

Use `lucide-react` exclusively. The mockup uses Lucide icons (`<i data-lucide="play">`); the React import is the camel-cased name (`import { Play } from "lucide-react"`).

The brand glyph is **not** Lucide. Use `<BrandGlyph size={22} />` from `@flink-reactor/ui`.

### Dynamic Tailwind class names — don't

Tailwind v4 scans source files statically. **Templated class strings like `` `text-${tone}` `` produce no utility.** Always use static strings, one per branch:

```tsx
// WRONG — Tailwind never generates these utilities
<span className={`text-${tone === "fail" ? "fr-rose" : "fr-sage"}`} />

// RIGHT
<span className={tone === "fail" ? "text-fr-rose" : "text-fr-sage"} />
```

This applies to Tailwind utilities (`text-*`, `bg-*`, `border-*`). It does **not** apply to custom CSS classes defined in `hub.css` like `.avatar-coral`, `.avatar-sage` — those work fine with templated names because the CSS rule already exists; Tailwind doesn't need to generate anything.

## Common pitfalls observed (don't repeat these)

| Pitfall | What goes wrong | Fix |
|---|---|---|
| Forgetting to add Hub color tokens to `dashboard/src/global.css` `@theme` | Color swatches transparent, `text-fr-sage` undefined, status icons invisible | Mirror tokens from `packages/ui/src/styles/tokens.css` into `global.css`'s `@theme` block |
| `var(--font-geist-mono)` instead of `var(--font-mono)` | Mono font silently falls back to body font | The dashboard's `@theme` defines `--font-mono`; use that exact name |
| Skipping the `@import "@flink-reactor/ui/styles/hub";` line in `global.css` | Hub CSS classes (`.nav-item`, `.btn`, `.cluster-selector`) don't load | Make sure the import is right after `@import "tailwindcss";` |
| Importing `hub.css` without `layer(components)` | Tailwind utilities (`pl-9`, `pr-4`, `text-fg-muted`) silently lose to hub.css class rules with the same specificity. Symptom: search icon overlaps placeholder, form padding wrong, etc. | Use `@import "@flink-reactor/ui/styles/hub" layer(components);` so Tailwind's `utilities` layer ranks above hub.css's `components` layer regardless of source order |
| Building a "good enough" Kanban (or any complex primitive) without reading the mockup | Output looks generic; missing version pills, resource bars, avatars, empty states, add-card footers | Read `console-v2/<page>.html` end-to-end; mirror card-by-card |
| Forgetting `aria-label` on icon buttons (theme/notifications/avatar) | Accessibility regression | Every icon-only button needs an `aria-label` |
| Hard-coding cluster info in routes | Each new route reinvents the cluster selector wiring | When the second route lands, lift cluster state into a store/hook and have `HubShell` consume it |
| Rendering raw `<a>` in sidebar items | Full page reloads on nav | Pass `LinkComponent` to `HubShell`; the sidebar uses it |

## Verification checklist (run before merging a Hub route)

- [ ] The page renders at `/hub/<path>` with `<HubShell>` chrome (top bar + sidebar + main).
- [ ] The sidebar highlights the active route (`activePath` prop is correct).
- [ ] Cluster selector shows the right env tint (prod=rose, stage=amber, dev=teal).
- [ ] All KPI / data values come from real GraphQL stores, not hardcoded.
- [ ] Where the mockup uses Geist Mono, the rendered text is in Geist Mono (verify via DevTools computed font).
- [ ] Status icons, priority bars, sev-badges all render with the correct color (not transparent — that means a missing token).
- [ ] No console errors about undefined CSS variables.
- [ ] Resize to 1280px: right rail collapses cleanly (or stacks below — never produces an empty side gap).
- [ ] Click around: navigation feels client-side (no full reloads).
- [ ] `pnpm --filter dashboard build` succeeds.
- [ ] `pnpm --filter @flink-reactor/ui build` succeeds.
- [ ] `pnpm test` — no new test failures.

Notable cross-cuts (P4 — `fr-console-hub-tools-instruments`):
- `<HubFlamegraph>` lives in `components/hub/tools/flamegraph/` and is consumed by both the JM Profiler tab and the TM-detail Profiler tab via `<ProfilerPicker>`.
- `<SchemaDiffViewer>` lives in `components/hub/tools/diff/` and composes the foundation `<DiffViewer>` over two parallel `schemaDetail` queries.
- `<SqlExplorer>` lives in `components/hub/tools/sql-explorer/` — a multi-statement console (`sql-explorer.tsx` shell + `results-tabs.tsx` per-statement tabs + `console-inspector.tsx`). It reuses `useCatalogExploreStore` for sequential multi-statement SQL Gateway execution (`executeAll`/`executeSelection`, reusing `splitStatements` from `sql-gateway-store`). The inspector deep-links statements that launched a job to `/hub/jobs/$id` (the Flink job id is surfaced via `SQLFetchResult.jobID`), with live state from the job store — deliberately *not* a link out to the stock Flink Web UI.
